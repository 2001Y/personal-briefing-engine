# Proactive Life Agent 構想詳細.md

> **For Hermes:** この文書は、Dawarich・カレンダー・メール・地図・SNSシグナルを統合し、ユーザーに先回りして提案と実行を行う『真の AI エージェント』の設計たたき台である。最小レイヤー・live データ取得・検証可能性を優先し、複雑化を避ける。

## Goal

位置変化・滞在・予定・受信イベント・購買ニーズをトリガーに、Hermes Agent が **自律的に状況判断し、提案し、必要なら実行直前まで進める** パーソナル実行エージェントを構築する。

## One-line Architecture

**「イベント収集 → 状況判定 → 候補調査 → 優先度付き提案/実行」の一本線を最小構成で実装し、トリガー自体も定期的に自己改善する。**

## Tech Stack / Building Blocks

- 位置: Dawarich
- エージェント本体: Hermes Agent
- スケジューラ: Hermes cronjob
- メッセージ到達先: Slack / 必要に応じて Telegram 等
- 調査: Web / Browser / 将来的な API 連携
- 予定: Google Calendar / Apple Calendar / ICS など
- メール起点: Gmail webhook or polling
- 地図文脈: Google Maps saved places / starred places / 将来的なエクスポート連携
- 話題性: X / Instagram / ブログ / ローカルレビュー
- 実行対象: レストラン提案、移動可否判断、買い物再購入、予約補助、通知

---

## 1. この構想で実現したい体験

ユーザーが明示指示しなくても、Hermes が次のように振る舞う。

1. 大きく場所が変わったことを検知する
2. その移動が『通過中』なのか『到着・滞在開始』なのかを判定する
3. 現在地・時刻・速度・次予定・過去嗜好から、今意味のある提案だけを作る
4. 近くのご飯・観光・休憩・買い物候補を収集する
5. 既存のユーザー文脈（Google Maps ピン、過去会話、SNS話題、営業時間、距離）で絞る
6. 「今行く価値が高いもの」だけを短く送る
7. 必要なら予約・購入・再注文・ナビ開始直前まで進める

重要なのは、単なる通知 bot ではなく、**文脈を踏まえて判断する実行エージェント** にすること。

## 2. 基本原則

### 2.1 最小レイヤー

ユーザーの好みに合わせ、過度な層分けは避ける。

- NG: event bus, workflow engine, recommendation microservice, policy service を最初から分ける
- OK: まずは 1 本の cron / runner が live データを取得し、判定し、必要時だけ Hermes を起動する

最初の構成は以下で十分。

- `collector`: 現在地・予定・受信・候補ソース取得
- `judge`: いま動くべきか判定
- `act`: 提案 or 実行

### 2.2 live データ優先

- 位置・速度・予定・営業時間・レビュー・SNS話題はできるだけ live 取得
- 推測より確認
- 「今開いているか」「今から間に合うか」「今混みそうか」を重視

### 2.3 実行は段階制

副作用の強い行為ほど段階を上げる。

- Level 0: 黙って観測・学習のみ
- Level 1: 提案だけ送る
- Level 2: 下書き・カート投入・予約画面遷移直前まで
- Level 3: ユーザー承認後に購入/送信/予約確定

### 2.4 トリガーも改善対象

トリガー条件は固定しない。

- どの通知が役立ったか
- 誤検知は何だったか
- 早すぎ/遅すぎたか
- どのイベントが空振りだったか

を定期的に見直し、閾値や条件を再提案させる。

## 3. 最初に扱うトリガー群

## 3.1 位置変化トリガー

最重要。Dawarich 由来の位置変化から開始する。

### 発火候補

- 前回観測点から一定距離以上移動
- 速度が高い状態から低い状態に落ちた
- 一定時間同一エリアに滞在した
- 普段行かないエリアに入った
- 既知 POI クラスタ付近に到着した

### 位置変化トリガーの狙い

- 食事候補提案
- 観光/寄り道候補提案
- 次予定への到着可否計算
- 近場の pinned place 再提示
- 帰宅/出社/出先モード判定

### 最初の現実的な判定式

以下のような素朴な式で十分に開始できる。

- 距離変化 > 800m
- 直近移動の平均速度が 20km/h 超から 5km/h 未満へ低下
- 現在地点が 8〜15 分以上ほぼ固定

この 3 条件が揃えば「どこかに着いた可能性が高い」とみなせる。

## 3.2 Gmail 受信トリガー

位置以外の高価値トリガー。

### 発火候補

- 予約確認メール
- 配送通知
- 今日/明日の予定変更
- 支払い・請求
- カレンダー招待
- EC 購入関連

### 活用例

- 「いま届いた予約確認メールの店、現在地から何分か」
- 「配送通知の商品を受け取れる場所にいるか」
- 「明日の予定変更を見て移動余裕を再計算」

## 3.3 カレンダー近接トリガー

### 発火候補

- 次予定が X 分以内
- 現在地から所要時間が閾値超過
- 出発推奨時刻を過ぎた
- 近くに余白時間が発生した

### 活用例

- 「今出ると 6 分遅れそう」
- 「会議まで 35 分空き、徒歩 7 分圏内で良さそうな店が 3 件ある」

## 3.4 買い物・消耗品トリガー

### 発火候補

- 過去会話で『欲しい』が言及された
- 過去購入から想定消費期間が経過
- Amazon / EC 価格が下がった
- 在庫切れしそうなカテゴリ

### 活用例

- トリートメント再購入
- 日用品補充
- 定番品の最安更新通知

## 4. 判断エンジンの最小設計

複雑な recommender system を最初から作らない。

毎回以下の 4 問に答えるだけでよい。

1. **今は提案すべきタイミングか?**
2. **何を提案すべきか?**
3. **提案だけでよいか、実行直前まで進めるか?**
4. **今この通知を送る価値は本当に高いか?**

### 入力

- current location
- recent movement history
- current speed / stop duration
- next calendar events
- local time / day of week
- saved places / favorites
- social buzz candidates
- email-derived events
- shopping backlog
- conversation memory

### 出力

- action type: `food_suggestion | sightseeing_suggestion | leave_now_warning | purchase_reminder | repurchase_prep | no_action`
- confidence
- urgency
- reasoning summary
- evidence links

## 5. レコメンド候補ソース

## 5.1 ご飯屋さん

優先順の基本案:

1. ユーザーが既に Google Maps でピンした店
2. 過去会話で好意的に言及した店やジャンル
3. 現在営業中で近い店
4. 直近 SNS で話題
5. レビュー評価が高い店

### 評価軸

- 現在地からの移動時間
- 営業中か
- 混雑しそうか
- ジャンル適合
- 一人/複数人適性
- 価格帯
- 雨天/屋内適性

## 5.2 観光・寄り道

- 現在地近辺のランドマーク
- 保存済みスポット
- Instagram / X で話題の展示・景観・イベント
- 滞在可能時間に収まるか

## 5.3 SNS 話題性

### X

- 地名 + グルメ名
- 地名 + カフェ
- 地名 + 展示/イベント
- 過去 24〜72 時間の言及密度

### Instagram

- エリアタグ
- 店舗名ハッシュタグ
- 直近投稿頻度
- 写真の雰囲気と混雑時間帯の推定

### 注意

SNS は話題性には強いが誇張やノイズも多いので、単独採用せず営業時間・地図・レビューと突き合わせる。

## 6. カレンダー統合の設計

最も価値が高いのは『現在地から本当に間に合うか』である。

### 必要データ

- 次イベントの開始時刻
- 開催場所 / URL / オンライン判定
- 移動手段候補
- 現在地からの概算移動時間
- バッファ時間

### 判定結果の種類

- 間に合う
- ギリギリ
- 遅れそう
- 既に危険
- オンラインなので移動不要

### 通知例

- 「19:00 の予定まで 42 分、現在地から電車で 31 分なので間に合います」
- 「徒歩では厳しいので 8 分以内に出発推奨です」
- 「予定場所近辺で 20 分以内に入れそうな店を 2 件添えます」

## 7. 買い物エージェント設計

今回は特に Amazon 再購入が対象。

### 目標

- ユーザーが以前使って良かった消耗品を再特定する
- 価格・容量・配送タイミングを確認する
- ユーザー承認の直前まで進める

### 再購入フロー

1. 過去会話・注文履歴・ブラウザ状態から商品特定
2. 同一商品か改良版かを確認
3. 容量違い・定期便・詰替え有無を確認
4. カート投入 or 購入確認画面直前まで進む
5. 最終確定はユーザー承認

### 買い物系の安全原則

- 別商品への勝手な置換をしない
- 香り違い・容量違い・詰替え/本体違いを明示
- 定期便は勝手に有効化しない
- 支払い確定は承認必須

## 8. トリガー見直しの自己改善ループ

ユーザー指摘どおり、ここが重要。

## 8.1 なぜ必要か

固定閾値はすぐ陳腐化する。

- 通勤・通学・出張で移動パターンは変わる
- 有効な通知タイミングは曜日や時間帯で変わる
- 「役立つ通知」の定義が生活リズムで変わる

## 8.2 最小の見直し方式

cron で週 1 回、以下を自己レビューさせる。

### 入力

- 直近 7〜14 日のトリガー発火一覧
- 発火後のユーザー反応
- 実際に採用された提案
- 無視された通知
- 誤検知と思われるケース

### 出力

- 閾値変更案
- 新規トリガー候補
- 削除すべきトリガー
- 通知文面改善案
- 実行レベル変更案

### 例

- 滞在判定を 10 分 → 14 分へ
- 夜はご飯提案、昼はカフェ提案を優先
- 駅付近では観光提案を抑制
- Gmail の配送通知は即時、広告系は無視

## 8.3 さらに良い発展

安定後は bandit 的な探索を導入できる。

- 提案タイミング A/B
- 候補ソース重み調整
- 通知長さ最適化

ただし最初からやりすぎない。

```json
    "lng": 139.0,
    "speed_kmh": 2.8,

  },
  "context": {

    "shopping_needs": ["treatment-repurchase"],

  }

```

DB を大きく設計しすぎず、まずは JSON ログ + 軽い集計で十分。

## 11. 運用上の重要論点

### 11.1 誤通知抑制

最重要。

- 移動中はあまり送らない
- 同一エリアで短時間に連投しない
- 深夜帯の通知抑制
- 予定が詰まっている時は提案より警告優先

### 11.2 プライバシー

- 位置履歴は必要最小限のみ処理
- 外部 API へ送る情報は縮約
- 共有先は自分だけを基本に
- 監査用ログは保持期間を短く

### 11.3 説明可能性

通知には根拠を短く添える。

- 「移動後 12 分滞在中」
- 「次予定まで 54 分」
- 「徒歩 6 分・営業中」
- 「保存済みスポット」

これにより、なぜ今それを提案したかが分かる。

## 12. 今回のトリートメント再購入タスクへの落とし込み

今回の単発タスクも、この構想の一部として扱える。

### 位置づけ

- 種別: `repurchase_prep`
- トリガー: ユーザー明示の購買ニーズ
- 期待動作: 商品特定 → 価格/容量確認 → 購入直前まで

### 今回の実務ステップ

1. 過去会話や Amazon 状態から最近使っているトリートメントを特定
2. 同一商品であることを確認
3. Amazon で再注文導線に入る
4. 最終確定直前で止める

## 13. 推奨する次アクション

優先順にやるとよい。

1. **Dawarich polling + 滞在判定 + Slack 提案** を最初の MVP として作る
2. 次に **Calendar 到着可否判定** を足す
3. その後 **shopping backlog / repurchase** を足す
4. 最後に **trigger self-review cron** を加える

この順番なら、最初から大規模化せず、価値の高い順に積める。

---

## 14. 成功判定

この構想が成功している状態は以下。

- 役立つ通知だけが届く
- 「今それ欲しかった」が増える
- 到着/滞在判定の誤爆が少ない
- 次予定への遅延警告が実用的
- 再購入や予約が手間なく進む
- トリガー精度が週ごとに改善する

---

## 15. まとめ

本構想の核心は、単に cron で何かを監視することではなく、**位置・予定・受信・購買意図を統合して、今この瞬間に意味のある行動だけを選ぶこと** にある。

そのためには、

- live データ取得
- 最小構成の判定器
- 実行レベルの段階制
- トリガー自体の自己改善

の 4 点を軸に進めるのがよい。

最初は小さく、しかし将来は十分に『真の AI エージェント』へ拡張できる構成になっている。
